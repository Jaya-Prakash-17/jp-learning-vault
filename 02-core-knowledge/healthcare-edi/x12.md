# ASC X12 Standard

**30-Second Explanation**: X12 is the specific dialect of EDI used in US Healthcare. It looks like a massive wall of text separated by asterisks (`*`) and tildes (`~`).

**2-Minute Explanation**: X12 is an incredibly dense, hierarchical data format. Instead of verbose XML tags (`<FirstName>John</FirstName>`), X12 uses short segments and elements (`NM1*IL*1*SMITH*JOHN`). This makes the files incredibly small, allowing clearinghouses to send millions of claims in a single batch file. The standard is maintained by the X12 committee, and HIPAA law legally mandates its use for healthcare transactions in the US.

**Purpose**: To provide a standardized, highly compressed data format for all US healthcare transactions.

**Who sends it**: Every system in the US healthcare revenue cycle.

**Who receives it**: Every system in the US healthcare revenue cycle.

**Input**: Relational database data mapped into X12 segments.

**Output**: A flat text file with strict segment delimiters (`~`) and element delimiters (`*`).

**Common Errors**: 
- Missing mandatory elements (e.g., an N3 segment without an address line).
- Incorrect delimiters causing the parser to crash.

**Real Project Example**: In the State of Colorado project, I used Edifecs MapBuilder to visually parse incoming X12 837I text files and map the hierarchical data (Loops, Segments, Elements) into a structured XML format the state's backend could process.

**Interview Questions**:
- *What are SNIP Levels in X12 validation?* (Answer: SNIP 1 is basic syntax/delimiters. SNIP 2 is structural/HIPAA rules. SNIP 3 is balancing, etc.)
- *What is the difference between a Segment and an Element?* (Answer: `NM1` is the segment. `SMITH` and `JOHN` are the elements within that segment).
