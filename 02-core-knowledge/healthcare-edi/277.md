# 277 Claim Status Response

**30-Second Explanation**: The answer to the 276. It tells the provider exactly where their claim is in the payer's system (e.g., Paid, Denied, Pending Review).

**2-Minute Explanation**: When a payer receives a 276, their integration engine queries the core adjudication system (like Facets). It takes the backend status and maps it to standardized STC (Status Information) segments in an X12 277 file. This ensures the provider's software can automatically read the response without human intervention. A 277 might say "Pending Medical Records" or "Denied: Member not eligible."

**Purpose**: To answer a 276 inquiry with machine-readable status updates, preventing call-center volume.

**Who sends it**: Payers (Insurance Companies).

**Who receives it**: Providers.

**Input**: A 276 Inquiry and the backend adjudication status.

**Output**: A parsed 277 X12 file sent back to the provider.

**Common Errors**: 
- Returning generic errors instead of specific STC codes, causing the provider to call anyway.
- Backend API timeouts causing the 277 generation to fail.

**Real Project Example**: For Humana, I designed the DataFlows that mapped internal JSON CORE API responses into X12 277 payloads. If the backend API timed out, I implemented an error-handling framework that caught the exception and generated a 277 with a specific STC code ("System Unavailable") rather than crashing the route.

**Interview Questions**:
- *What is the difference between a 277 and an 835?* (Answer: 277 is just a status update. 835 is the final Remittance Advice that is tied to actual money hitting the bank).
